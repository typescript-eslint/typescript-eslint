`void` in TypeScript refers to a function return that is meant to be ignored.
The `void` operator is a useful tool to convey the programmer's intent to intentionally not await a Promise-like value.
For example, it is recommended as one way of suppressing [`@typescript-eslint/no-floating-promises`](./no-floating-promises.mdx) instead of adding `.catch()` to a promise.

This rule helps authors catch API changes where async work was previously being marked as intentionally not awaited at a call site, but the callee changed so it no longer returns a Promise-like value.
When combined with [no-unused-expressions](https://eslint.org/docs/rules/no-unused-expressions), it also helps _readers_ of the code by ensuring consistency: a statement that looks like `void foo();` is **always** marking a possibly async result as intentionally not awaited, and a statement that looks like `foo();` is **never** doing so.
This rule reports on any `void` operator whose argument is already of type `void` or `undefined`.
It also reports on `void` expression statements whose argument is not possibly Promise-like.

## Examples

<Tabs>
<TabItem value="❌ Incorrect">

```ts
declare const box: { value: string };

void box;
void box.value;
void box.value.toUpperCase();

void (() => {})();

function foo() {}
void foo();

declare function returnsString(): string;
void returnsString();
```

</TabItem>
<TabItem value="✅ Correct">

```ts
declare const box: { value: string };

box;
box.value;
box.value.toUpperCase();

(() => {})();

function foo() {}
foo(); // nothing to discard
```

## When Not To Use It

If you don't mind extra `void`s in your project, you can avoid this rule.
