TypeScript supports defining arbitrary object keys using an index signature or mapped type.
TypeScript also has a builtin type named `Record` to create an empty object defining only an index signature.
For example, the following types are equal:

```ts
interface IndexSignatureInterface {
  [key: string]: unknown;
}

type IndexSignatureType = {
  [key: string]: unknown;
};

type MappedType = {
  [key in string]: unknown;
};

type RecordType = Record<string, unknown>;
```

Using one declaration form consistently improves code readability.

## Options

- `'record'` _(default)_: only allow the `Record` type.
- `'index-signature'`: only allow index signatures.

## FAQs

### What are the differences between `Record` and an index signature?

There are a few practical differences between the two forms for defining indexed object types.

:::note
This rule treats index signatures (`{ [key: string]: V }`), mapped types over primitive keys (`{ [K in string]: V }`), and `Record<string, V>` as equivalent.
The differences below apply between the `Record<K, V>` shorthand and the verbose index signature form.
:::

| Feature                     | `Record<K, V>`                          | Index Signature `{ [key: K]: V }`                                     |
| --------------------------- | --------------------------------------- | --------------------------------------------------------------------- |
| Conciseness                 | ✅ Shorter                              | More verbose                                                          |
| Mixed with named properties | Via intersection: `Record<K,V> & {...}` | ✅ Directly within the type definition                                |
| Named key parameter         | ❌                                      | ✅ Allows naming the key for documentation (e.g., `[userId: string]`) |
| Union/literal key types     | ✅ `Record<'a' \| 'b', V>`              | ❌ Not supported — index signatures only accept `string`, `number`, `symbol`, and template literals as key types |
| `readonly` modifier on value | Via `Readonly<Record<K, V>>`           | ✅ `{ readonly [key: K]: V }`                                         |

#### Mixing with named properties

Index signatures allow combining named and indexed properties directly within a single type:

```ts
interface UserMap {
  default: User;
  [userId: string]: User;
}
```

With `Record`, you can achieve the same result using an intersection type:

```ts
type UserMap = Record<string, User> & { default: User };
```

#### Union and literal key types

`Record<K, T>` is defined as the mapped type `{ [P in K]: T }`, so `Record<'featureA' | 'featureB', boolean>` and `{ [K in 'featureA' | 'featureB']: boolean }` are semantically identical.
Both create a type with required properties for every key in the union:

```ts
type Flags = Record<'featureA' | 'featureB', boolean>;
// Identical to: { [K in 'featureA' | 'featureB']: boolean }

// Error: Property 'featureB' is missing in type '{ featureA: true }'.
const flags: Flags = { featureA: true };
```

Index signatures, by contrast, only accept primitive key types (`string`, `number`, `symbol`, and template literals).
Union and literal key types are not valid in an index signature:

```ts
// ❌ Not valid TypeScript — union types are not allowed in index signatures
type Flags = { [key: 'featureA' | 'featureB']: boolean };

// ✅ Use a mapped type or Record instead
type Flags = { [K in 'featureA' | 'featureB']: boolean };
type Flags = Record<'featureA' | 'featureB', boolean>;
```

We recommend choosing one style and applying it consistently.
The only times you **must** use an index signature over `Record` are when you need to name the key parameter for documentation purposes (e.g., `[userId: string]`), or when combining named and indexed properties within a single type definition.

## When Not To Use It

This rule is purely a stylistic rule for maintaining consistency in your project.
You can turn it off if you don't want to keep a consistent style for indexed object types.

However, keep in mind that inconsistent style can harm readability in a project.
We recommend picking a single option for this rule that works best for your project.
